Foundation · Week 1
Linear Regression
The first supervised learning model — and the building block of every neural network ever built

What is supervised learning? You give the algorithm a dataset of input-output pairs: house sizes and their sale prices. The algorithm learns a mapping function f from input x to output y. Once trained, it predicts prices for new houses it has never seen.

The model: Linear regression assumes this mapping is a straight line. The model is f(x) = wx + b where w (weight) controls the slope — how much price changes per unit of size — and b (bias or intercept) is the baseline price when size = 0. Together, w and b are the model's parameters, and training means finding the best values for them.

Why "linear"? The model output is a linear function of the parameters w and b. Doubling w doubles the prediction change. This linearity makes the math tractable and the optimization guaranteed to work — but it's also a limitation. Real-world relationships are rarely perfectly linear, which is why we later extend to polynomial features and neural networks.

The notation unpacked: x = input feature (house size). y = true output (actual price). ŷ = predicted output from the model. m = number of training examples. Superscripts like x⁽ⁱ⁾ denote the i-th training example.

Model: f_w,b(x) = w·x + b w = slope (weight) → how output changes per unit of input b = intercept (bias) → output when x = 0 Training: find w,b that minimize prediction errors across all (x⁽ⁱ⁾, y⁽ⁱ⁾)

One neuron: A single neuron in a neural network computes exactly output = w·x + b — and then passes the result through a non-linear activation function (like ReLU or sigmoid). Linear regression is a neuron without the activation. Stack hundreds of neurons in layers, and you get deep learning. This is not metaphorical — it is exactly the same math.

📐
w = slope, b = intercept
If w=1.5: a house 1 unit larger costs 1.5 units more. If b=0.5: the baseline price for size=0 is 0.5. Changing w rotates the line; changing b shifts it up/down.
🎯
Prediction vs Truth
The error (residual) for each point is ŷᵢ - yᵢ. The red lines in the tool show these errors. Training means making all red lines as short as possible — simultaneously.
🧠
Linear Reg = 1 Neuron
Neural network layer: output = W·X + b (matrix form). Same formula. GPT-4 runs this billions of times per forward pass — just with much larger matrices.
🤖 In Production AI
Every transformer layer in GPT-4, BERT, DALL·E is: output = W·input + b. This is matrix linear regression. The "weights" W are what gradient descent learns during training. Linear regression is not a toy — it IS deep learning, scaled up.

Measurement · Week 1
Cost Function J(w,b)
The single number that tells the model how wrong it is — and defines what "learning" even means
❓
Why square the errors?
Three reasons: (1) makes all errors positive — negative errors don't cancel positive ones. (2) penalizes large errors heavily — a mistake of 2 units costs 4×, not 2×. (3) creates a smooth, differentiable curve — essential for gradient descent to work.
½
The 1/(2m) factor
Dividing by m: averages cost across examples so it doesn't grow with dataset size. The extra ½: purely mathematical convenience — it cancels with the 2 that appears when you differentiate x², giving cleaner gradient formulas. It doesn't change where the minimum is.
🏔️
Convexity — the key guarantee
With MSE + linear regression, J(w,b) is always convex — a perfect bowl with ONE minimum. Gradient descent is guaranteed to find the global minimum. Deep neural networks lose convexity — optimization is much harder.
📊
2D vs 3D view
Toggle between views in the tool. The 1D curve (J vs w, b fixed) is a parabola. The 2D contour (J vs both w and b) shows a bowl viewed from above — the tighter/inner ellipses are lower cost. The center = minimum J.
🤖 Loss functions in AI
MSE is just one of many. The loss defines the task: cross-entropy for classification (GPT predicts next token), CLIP loss for image-text matching (DALL·E), RLHF reward for ChatGPT alignment. Choosing the right loss = defining what the model should optimize for.
MSE → regressionCrossEntropy → GPTCLIP → DALL·ERLHF → ChatGPT

Why do we need a cost function? "Training" means improving the model. But improving means moving toward something. The cost function defines what we're trying to minimize — it's the objective. Without it, gradient descent has no direction to move.

Mean Squared Error (MSE): For each training example, compute the prediction error ŷᵢ - yᵢ. Square it (to make it positive and penalize large errors). Average over all m examples. Divide by an extra 2 for convenience. The result is J(w,b) — a single number measuring total model error.

J(w,b) = 1/(2m) · Σᵢ₌₁ᵐ (f_w,b(xᵢ) - yᵢ)² = 1/(2m) · Σᵢ₌₁ᵐ (w·xᵢ + b - yᵢ)² m = number of training examples (wᵢ·xᵢ + b) = prediction (ŷᵢ) yᵢ = actual value (ŷᵢ - yᵢ) = error (residual) for example i

The cost surface shape: J(w,b) forms a bowl-shaped surface in 3D (w, b, J axes). Every combination of w and b gives a different J. The center of the bowl — the minimum — is the optimal w★ and b★ we're searching for. In 2D contour view, concentric ellipses show equal-cost contours, like elevation lines on a topographic map.

J is always ≥ 0. Since it's a sum of squares divided by a positive number, J can never be negative. J = 0 only if every prediction is perfect — which happens only on noiseless data. In practice, we aim for J as small as possible, not exactly 0.

Optimization · Week 1
Gradient Descent
The algorithm that trains every AI model ever created — from linear regression to GPT-4

The core idea — roll downhill: Imagine you're standing on a hilly landscape (the cost surface) blindfolded. You want to reach the valley (minimum J). The only information you have is the slope of the ground beneath your feet (the gradient). So you take a small step downhill. Then repeat. Eventually you reach the valley. This is gradient descent.

The update rule: At each step, compute the partial derivative of J with respect to each parameter. The partial derivative is the slope — positive means J increases in that direction, so we move the opposite way. The step size is controlled by α (alpha), the learning rate. Both parameters are updated simultaneously using the old values — not sequentially.

Simultaneous update (critical!): temp_w = w - α · ∂J/∂w temp_b = b - α · ∂J/∂b w = temp_w ← update AFTER computing both b = temp_b ← using OLD values of w and b Partial derivatives: ∂J/∂w = (1/m) · Σ(wxᵢ + b - yᵢ) · xᵢ ∂J/∂b = (1/m) · Σ(wxᵢ + b - yᵢ)

Why simultaneous? If you update w first and then use the new w to compute the gradient for b, you'd be computing ∂J/∂b at the wrong point — the math is inconsistent. Both derivatives must be computed at the same (w, b) position before any update happens. This is a subtle but important implementation detail.

Convergence: Gradient descent converges when J stops decreasing meaningfully. Near the minimum, the gradient (slope) approaches zero, so the steps become tiny. The algorithm naturally slows down as it approaches the optimum — it doesn't overshoot when α is chosen correctly.

Backpropagation is gradient descent: In neural networks, ∂J/∂w must be computed for every weight — possibly billions of them. Backpropagation uses the chain rule to efficiently propagate gradients backward through the network. The update rule is identical: w := w − α·∂J/∂w. The algorithm is the same; only the computation graph is more complex.

∂
What ∂J/∂w means
The partial derivative is the slope of J in the w direction — if you increase w by a tiny amount ε, J changes by ≈(∂J/∂w)·ε. When positive: increasing w increases J, so decrease w. When negative: decrease w increases J, so increase w. Always move opposite the gradient.
⚠️
Why simultaneous update matters
Wrong: update w, then compute ∂J/∂b with new w. Right: compute both derivatives with old w and b, then update both. Sequential updates use an inconsistent gradient — different from what you'd get mathematically.
🏁
Batch vs Mini-batch vs SGD
Batch GD: use all m examples to compute gradient (slow per epoch, stable). Stochastic GD (SGD): use 1 example at a time (fast, noisy). Mini-batch GD: use 32-256 examples (the standard in deep learning — fast + stable). GPT uses mini-batch GD with batch size ~1M tokens.
🤖 Adam — gradient descent evolved
Modern AI uses Adam optimizer — it keeps a running estimate of the gradient mean (momentum) and squared gradient (variance) for each weight separately. This gives each parameter its own effective learning rate that adapts over time. Nearly every foundation model (GPT, BERT, LLaMA) trains with Adam.

Hyperparameter · Week 1
Learning Rate α
The step size that determines whether gradient descent succeeds, crawls, or explodes
🐌
α too small → glacially slow
Each step reduces J by a tiny amount. The algorithm is mathematically correct and will converge — but it might take 100,000 iterations instead of 1,000. In practice, training might time out before converging. With very small α, you might also get stuck near a saddle point.
✅
α just right → smooth convergence
J decreases steadily every iteration. The learning curve is smooth. Convergence happens in a reasonable number of steps. The "right" α depends on the problem — typically found by grid search or by watching the learning curve and adjusting by factors of 3–10.
💥
α too large → divergence
Each step overshoots the minimum. J bounces back and forth, getting larger each time. Eventually J → ∞. Debugging tip: if J ever increases during training, your α is too large. Reduce by 10× and try again.
🔬
How to choose α in practice
Try: 0.001, 0.003, 0.01, 0.03, 0.1, 0.3, 1.0 (roughly 3× increments). Plot J vs iterations for each. Pick the largest α that still decreases J smoothly. Andrew Ng recommends this systematic approach over guessing.
🤖 How modern AI handles learning rate
Adam optimizer adapts each weight's effective LR based on gradient history — you rarely tune α manually. But the initial α (base LR) still matters. GPT-4 uses linear warmup (α starts near 0, grows for first ~2000 steps) then cosine decay (α smoothly drops to 0 over training). This prevents instability early and ensures fine convergence at the end.
LR WarmupCosine DecayCyclical LRAdam / AdamW

What α controls: α (alpha) is the step size in gradient descent. After computing the gradient ∂J/∂w, we move by α × gradient in the opposite direction. Large α = large steps. Small α = small steps. The gradient only tells us the direction; α tells us how far to go.

w := w - α · ∂J/∂w α too small: takes forever to reach minimum α too large: overshoots minimum, diverges α just right: converges in reasonable iterations Debugging rule: If J increases at any step → α is too large. Reduce by 10×. If J decreases but very slowly → try increasing α by 3×.

Near the minimum, steps automatically shrink. Even with a fixed α, gradient descent naturally takes smaller steps near the minimum because the gradient (slope) becomes smaller near the flat bottom of the bowl. You don't need to manually reduce α as you get close — the math handles it. This is different from gradient clipping or schedules.

The 3× rule: When searching for a good α, try values spaced roughly 3× apart: 0.001, 0.003, 0.01, 0.03, 0.1, 0.3. This covers many orders of magnitude efficiently. Plot J vs iterations for each — pick the fastest-converging without diverging. Modern optimizers (Adam) are less sensitive to α choice.

Learning rate schedules: In deep learning, using a fixed α throughout training is suboptimal. Common schedules: cosine decay (α smoothly drops to 0 following a cosine curve), step decay (reduce α by factor 0.1 every N epochs), warmup + decay (used for transformers). The right schedule can significantly improve final model quality.

Vectorization · Week 2
Multiple Features & Vectorization
Extending to n inputs — and why a one-line numpy call runs 1000× faster than a Python loop

Real problems have many features. House price depends on size AND bedrooms AND floors AND age. Weather prediction uses thousands of sensor readings. Language models process sequences of thousands of tokens. We need to extend linear regression to handle n input features.

The model with n features: Replace single input x with a feature vector x⃗ = [x₁, x₂, ..., xₙ] and a weight vector w⃗ = [w₁, w₂, ..., wₙ]. The prediction becomes the dot product plus bias. Each wⱼ represents how much feature j affects the output.

Model (n features): f_w,b(x⃗) = w₁x₁ + w₂x₂ + ... + wₙxₙ + b = w⃗ · x⃗ + b (dot product notation) Gradient descent (each weight wⱼ): ∂J/∂wⱼ = (1/m) · Σᵢ (ŷᵢ - yᵢ) · xⱼ⁽ⁱ⁾ wⱼ := wⱼ - α · ∂J/∂wⱼ (for j = 1,...,n simultaneously) b := b - α · ∂J/∂b

Vectorization — the speed secret: A Python loop computing sum(w[j]*x[j] for j in range(n)) runs one multiplication at a time — O(n) sequential operations. NumPy's np.dot(w, x) calls optimized BLAS routines that use CPU SIMD instructions (doing 8+ operations simultaneously) or GPU CUDA (thousands of parallel cores). Same mathematical result. 1000× faster.

Matrix form for all training examples: Stack all m training examples into matrix X (m×n). The prediction for all examples at once: Ŷ = X·w⃗ + b. This is a single matrix multiplication — one GPU operation regardless of m and n. The entire deep learning field runs on this insight.

# Python loop — sequential, slow pred = sum(w[j]*x[j] for j in range(n)) + b # NumPy — parallel, SIMD/GPU accelerated pred = np.dot(w, x) + b # identical result # All m examples at once (matrix form) Ŷ = X @ w + b # X is (m×n), w is (n,)
🧮
Dot product = weighted sum
w⃗·x⃗ = w₁x₁ + w₂x₂ + ... + wₙxₙ. Each weight wⱼ scales how much feature j matters. Negative weights mean "more of this feature → lower prediction." Zero weight means "this feature is ignored by the model."
⚡
Why vectorization matters at scale
GPT-4 processes 1,024 tokens simultaneously, each a 768-dim vector. That's 1,024×768 ≈ 786K numbers per attention head, times 96 heads, times 96 layers = 72 billion operations per forward pass. Only possible with GPU matrix multiplication.
📐
Gradient descent scales naturally
The update rule for each wⱼ is the same structure as before — just applied to n weights. In matrix form: w := w - α · Xᵀ(Xw-y)/m. Same algorithm, vectorized. One line updates all n weights simultaneously.
🤖 Transformers = stacked matrix multiplications
Every attention head in GPT: Attention(Q,K,V) = softmax(QKᵀ/√d)·V. Q, K, V are each produced by linear transformations (matrix multiplies) of the input. 96 heads × 96 layers = 9,216 matrix multiplications per forward pass — all running in parallel on GPU.

Preprocessing · Week 2
Feature Scaling
Why features on different scales cause gradient descent to zig-zag — and how to fix it
🐍
The zig-zag problem
If x₁ ∈ [0,3000] and x₂ ∈ [0,5], the cost surface is an elongated ellipse. Gradient descent must take many tiny steps in the x₁ direction and few large steps in the x₂ direction — it zig-zags back and forth, taking far more iterations than necessary.
⚖️
Three scaling methods
Min-max: x = (x-min)/(max-min) → [0,1]
Z-score: x = (x-μ)/σ → mean=0, std=1
Mean norm: x = (x-μ)/(max-min) → [-1,1]

Z-score is most common — handles outliers better.
⚠️
Critical: fit on train, apply to test
Compute μ and σ using only the training set. Then apply the same transformation to validation and test sets. Never compute scaling statistics on test data — that's data leakage, a common mistake that inflates test performance.
🏗️
Batch Norm = scaling inside the network
Batch Normalization (2015) applies normalization to activations inside the neural network — after every matrix multiply. It enabled training of much deeper networks (ResNets with 100+ layers). The architecture: Linear → BatchNorm → ReLU → next layer.
🤖 LayerNorm powers every transformer
GPT, BERT, LLaMA all use Layer Normalization — normalizes each token's activation vector independently (unlike BatchNorm which normalizes across the batch). Without LayerNorm, training large transformers is unstable. RMSNorm (used in LLaMA/Mistral) is an even simpler variant that just divides by RMS.
BatchNorm → CNNsLayerNorm → GPT/BERTRMSNorm → LLaMAGroupNorm → Detection

The core problem: Gradient descent uses the gradient direction to navigate the cost surface. When features have very different scales, the cost surface is highly elongated — like a narrow canyon instead of a circular bowl. The gradient points across the canyon, not down it, causing the optimizer to zig-zag and converge slowly.

The solution — normalize features: After scaling, all features have comparable ranges. The cost surface becomes roughly circular (for 2 features), and gradient descent can take efficient straight-line paths to the minimum. In practice, this can reduce training time from hours to minutes.

Z-score normalization (most common): x_norm = (x − μ) / σ μ = mean(x_train) σ = std(x_train) → normalized feature has mean=0, std=1 Apply same μ, σ to test/validation data: x_test_norm = (x_test − μ_train) / σ_train ← not μ_test!

When is scaling necessary? Always recommended when: features have different units (size in sqft vs bedrooms), feature ranges differ by more than 10×, or using gradient-based optimization. Tree-based models (random forest, XGBoost) don't need feature scaling — they split on individual features independently.

The impact on cost surface shape: Without scaling, J(w₁,w₂) contours are stretched ellipses. The gradient is roughly perpendicular to the long axis, pointing across the canyon rather than toward the minimum. After scaling, contours become roughly circular, and the gradient points much more directly toward the minimum — faster convergence.

Monitoring · Week 2
Checking Gradient Descent Convergence
How to read learning curves — detecting when training is working, stuck, or broken

The learning curve: Plot J(w,b) on the y-axis against the number of training iterations on the x-axis. This "learning curve" is the primary tool for diagnosing gradient descent. A healthy learning curve decreases monotonically and eventually flattens — never increases.

Declaring convergence: Gradient descent has converged when J decreases by less than ε per iteration. Andrew Ng recommends ε = 10⁻³. Alternatively: run for a fixed number of iterations and check the final J against a baseline. The "convergence number" varies by problem — some need 1,000 iterations, others 1,000,000.

Convergence criterion: |J(iteration i) - J(iteration i-1)| < ε where ε = 10⁻³ (typical) Diagnostic rules: J decreasing monotonically → α is OK, keep training J oscillates → α too large, reduce by 3-10× J increases → α way too large OR bug in code J flat from start → α too small OR gradient vanishing

Training vs Validation curves: The training loss J_train measures performance on training data. The validation loss J_val measures performance on unseen data. The gap between them diagnoses overfitting. A healthy model has J_val tracking J_train closely. If J_train ≪ J_val, the model is memorizing training data — it won't generalize.

Early stopping: When J_val stops decreasing (or starts increasing), training has reached the optimal generalization point. Save the model weights from this iteration and stop training. This simple technique prevents overfitting without explicit regularization — the model never gets a chance to over-memorize training data.

What different shapes mean: A flat start indicates the learning rate may be too small, or the model may be stuck on a plateau. An early peak in J_val followed by continuous decrease in J_train is the signature of overfitting beginning. Oscillations in J_train suggest α is near the divergence threshold.

✅
Good convergence pattern
J decreases smoothly and monotonically. The decrease rate slows down (logarithmic shape). Eventually J flattens — that's convergence. Both J_train and J_val are close together and decreasing.
🔁
Overfitting signature
J_train keeps decreasing. J_val decreases for a while, then stops decreasing or starts increasing. The growing gap between them is overfitting — the model learned the training noise, not the underlying pattern.
📏
Underfitting signature
Both J_train and J_val are high and plateau together (with a small gap). The model is too simple — it can't fit even the training data well. Fix: increase model complexity (more features, more neurons) or train longer.
🛑
Early stopping
Monitor J_val every epoch. When J_val stops improving for k consecutive epochs (patience=k), stop training and restore weights from the best epoch. k=5–20 is typical. This is regularization for free — no extra hyperparameters except patience.
🤖 TensorBoard monitors real AI training
Every production ML training run logs loss curves to TensorBoard, Weights & Biases, or MLflow. Training GPT-4 costs ~$100M in compute. Engineers watch the loss curve in real-time — catching a diverging loss at hour 2 vs hour 200 could save millions of dollars and weeks of GPU time.

Feature Engineering · Week 2
Polynomial Regression & Feature Engineering
How adding x², x³ as features fits curves — and why neural networks made this obsolete
🌀
Still linear regression!
Polynomial regression with features [1, x, x², x³] is still linear regression — just with engineered features. The model is linear in the parameters w even if non-linear in x. The same gradient descent algorithm works unchanged.
📏
Feature scaling is critical here
If x ∈ [0,1000], then x² ∈ [0,10⁶] and x³ ∈ [0,10⁹]. These wildly different scales will make gradient descent fail without normalization. Always apply z-score normalization BEFORE adding polynomial features.
🎯
Bias-variance tradeoff
Low degree (1-2): High bias — model too simple, underfits. Both train and test error are high. Fix: increase complexity.

High degree (8+): High variance — model memorizes training noise, overfits. Train error low, test error high. Fix: regularization or reduce complexity.
🔬
Choosing the degree
Use cross-validation: split data into train/validation. Train models of degree 1, 2, ..., 10. Pick the degree with lowest validation error. Never use test data to select the degree — that's data leakage.
🤖 Neural nets = automatic feature engineering
Manual polynomial features (x², x³, sin(x), log(x)) require human expertise. Neural networks learn features automatically — each hidden layer creates new non-linear combinations of the previous layer's features. The Universal Approximation Theorem guarantees a sufficiently large single hidden layer can approximate any continuous function.
Universal ApproximationRepresentation LearningAutomatic FeaturesTransfer Learning

When linear isn't enough: Some relationships are inherently curved. House prices might scale faster than linearly with size (larger houses cost disproportionately more per sqft). A straight line can't fit this — we need a curve. Polynomial regression adds polynomial powers of x as new features to capture non-linear patterns.

Polynomial features from single input x: x₁ = x x₂ = x² x₃ = x³ x₄ = √x Model becomes: f(x) = w₁x + w₂x² + w₃x³ + b = w⃗ · [x, x², x³] + b Still linear regression — linear in parameters w! Same gradient descent, same cost function, same update rules.

The bias-variance tradeoff: This is the fundamental tension in all of machine learning. Increase model complexity (higher degree) → lower bias (fits training data better) but higher variance (sensitive to training noise, generalizes poorly). Decrease complexity → higher bias but lower variance. The optimal point is found through cross-validation, not by looking at training error alone.

Regularization — the cure for overfitting: Instead of restricting the degree, we can add a penalty term to the cost function that discourages large weights. L2 regularization (ridge): J = MSE + λ·Σwⱼ². L1 regularization (lasso): J = MSE + λ·Σ|wⱼ|. The hyperparameter λ controls the trade-off between fit and simplicity. This allows using high-degree polynomials without overfitting.

Why neural nets replaced feature engineering: Polynomial feature engineering requires domain expertise — which features to create, which degree to use. Neural networks learn a hierarchy of non-linear features automatically from data. The first layer might learn edges, the second layer shapes, the third layer objects — all from raw pixels with no human-designed features. This is why deep learning outperforms classical ML on raw data (images, text, audio).
